# Fitbod schema research

_2026-04-25 — pre-MITM. Update after capture confirms the live shape._

## Sources consulted

| Source | What it is | Signal level |
| --- | --- | --- |
| [Fitbod/my_workout](https://github.com/Fitbod/my_workout) | A Rails sample/interview project from Fitbod's GitHub org | First-party but **sample, not live** |
| [rrebase/fitdata](https://github.com/rrebase/fitdata) | Third-party visualizer for Fitbod's official user-facing CSV export | Third-party, but parses real exports |
| [Fitbod/parse-server](https://github.com/Fitbod/parse-server) | Fork of `parse-community/parse-server` | Confirms Parse Server is the backend |

The two evidence sources disagree on field-name style (snake_case in the
Rails sample vs. camelCase in the CSV export). The CSV is generated by the
live app, so it's the stronger signal for live Parse field names.

## Class names — high confidence

`my_workout`'s `app/models/{workout,single_set,exercise}.rb` defines:

- `Workout belongs_to :user, has_many :single_sets`
- `SingleSet belongs_to :workout, belongs_to :exercise`
- `Exercise has_many :single_sets`

Parse Server convention is PascalCase, so the live class names are almost
certainly:

- `_User` (Parse system class)
- `Workout`
- `SingleSet`
- `Exercise`

## Field names — medium confidence

**Rails sample** (`db/schema.rb`, snake_case — likely _not_ the live shape):

```
workouts:    user_id, workout_date, workout_duration
single_sets: weight (float), reps, performed_at, workout_id, exercise_id
exercises:   name
```

**Fitbod's CSV export** — strongest signal. From `rrebase/fitdata`'s
`Row` interface in `src/features/results/Results.tsx`:

```ts
interface Row {
  date: string;
  exercise: string;
  reps: number;
  weight: number;
  duration: number;
  distance: number;
  incline: number;
  resistance: number;
  isWarmup: string;
  note: string;
}
```

The CSV is a denormalized join (one row per set, exercise name inlined).
The column names should mirror the underlying Parse field names.

**Best guess for live Parse fields** (verify during MITM):

```
SingleSet:  weight, reps, duration, distance, incline, resistance,
            isWarmup, note, date (or performedAt),
            workout (Pointer<Workout>), exercise (Pointer<Exercise>)
Workout:    date (or startTime), duration, name?, user (Pointer<_User>)
Exercise:   name
```

## Units — ⚠️ kg, not lbs

`fitdata`'s UI labels its weight axis as **kg**. Fitbod's CSV export weight
is therefore in kg.

This codebase currently uses `WeightLbs` and `total_volume_lbs`. If the
live data comes out in kg, either:

- Convert at the normalize boundary (multiply by 2.20462 in
  `sync/fitbod/normalize.go`), or
- Rename internal fields and present stats in kg.

## Fields likely missing from the live data

Things this codebase assumes but the public sources don't show:

- **`e1rm`** — almost certainly computed client-side. The Epley fallback
  in `sync/stats/headline.go` already handles missing values.
- **`isCompleted`** — not in the CSV. Probably implicit (any exported set
  is considered completed). Treat absence as `true` in normalize.
- **`isCompound`** — not in Fitbod's data anywhere we can see. Two options:
  drop the "compound preferred" heuristic, or maintain a hand-curated
  list of compound exercise names.

## Open questions for the MITM session

1. **Class names:** PascalCase (`Workout`) vs snake_case (`workouts`)?
   Parse defaults to the former; confirm.
2. **Weight unit on the wire** — kg or lbs?
3. **Date field on `Workout`** — `date`, `startTime`, `performedAt`, or
   `workoutDate`?
4. **e1RM** — stored or always computed in-app?
5. **Compound flag** — is there an `isCompound` (or similar) on
   `Exercise`, or is the "compound vs isolation" distinction UI-only?
6. **Pagination** — how does the app fetch history? Bulk on login,
   paginated, or some live-sync protocol?
